What are Salesforce Governor Limits? Give the most important ones by number. APEX

Governor limits are runtime limits enforced by the Salesforce platform to ensure no single Apex execution consumes too many shared resources across the multitenant environment. These limits apply per transaction — meaning per trigger execution, per Apex class call, per batch execute() chunk, etc.

Critical limits to memorize:

  • SOQL queries per transaction: 100 (synchronous), 200 (async)
  • DML statements per transaction: 150
  • Records per DML statement: 10,000
  • Total heap size: 6 MB (sync), 12 MB (async)
  • CPU time: 10,000 ms (sync), 60,000 ms (async)
  • SOSL queries per transaction: 20
  • Callouts per transaction: 100
  • Records returned by SOQL: 50,000
  • Records returned by Database.QueryLocator (Batch): 50,000,000
  • Future methods per transaction: 50
  • Queueable jobs added per transaction: 50
  • Email invocations: 10
How to answer this in interview

Don't just list numbers — explain WHY limits exist. Say: "Salesforce is a multitenant platform. If one customer's code ran unlimited SOQL queries, it would slow down all other customers on the same server. Governor limits are Salesforce's way of enforcing fair resource usage across all tenants."

What is bulkification and how do you implement it correctly? APEX

Bulkification means writing Apex code that handles batches of records efficiently instead of processing one record at a time. This is critical because Salesforce can pass up to 200 records to a trigger in a single execution. If your code runs a SOQL query inside a loop over 200 records, you hit 200 SOQL calls — which exceeds the 100-query limit.

The anti-pattern (WRONG — will fail in production):

// BAD CODE - DO NOT DO THIS
for(Contact c : Trigger.new) {
    // SOQL inside loop = governor limit error!
    Account acc = [SELECT Id, Name FROM Account WHERE Id = :c.AccountId];
    c.Description = acc.Name;
}

The correct bulkified pattern:

// GOOD CODE - Bulkified properly
// Step 1: Collect all IDs into a Set
Set<Id> accountIds = new Set<Id>();
for(Contact c : Trigger.new) {
    if(c.AccountId != null) accountIds.add(c.AccountId);
}

// Step 2: ONE SOQL query outside the loop
Map<Id, Account> accMap = new Map<Id, Account>(
    [SELECT Id, Name FROM Account WHERE Id IN :accountIds]
);

// Step 3: Loop using Map - no SOQL inside loop
for(Contact c : Trigger.new) {
    if(accMap.containsKey(c.AccountId)) {
        c.Description = accMap.get(c.AccountId).Name;
    }
}
Key rule to remember

Collect in a Set → Query once with IN clause → Process from a Map. This pattern: one SOQL query handles 200 records. Never put SOQL or DML inside a for loop over sObjects.

When asked this, always give a concrete real-world use case. Say: "I use Set to collect Account IDs from a list of Contacts before querying, because IN clause requires a Set. Then I put query results in a Map for fast lookup inside a processing loop."

What are the different types of Apex exceptions and how do you handle them? APEX

Common built-in exception types:

  • DmlException — thrown when DML operation fails (required field missing, duplicate rule, etc.)
  • QueryException — thrown when SOQL returns no records for assignment to single sObject
  • NullPointerException — accessing property on null variable
  • LimitException — governor limit exceeded (cannot be caught)
  • CalloutException — HTTP callout failure, timeout
  • SObjectException — field not in query accessed on sObject
  • TypeException — invalid type cast
// Always use specific exception types, not generic Exception
try {
    Account a = [SELECT Id FROM Account WHERE Name = 'Test' LIMIT 1];
    insert a;
} catch(QueryException qe) {
    // No records returned by SOQL
    System.debug('No account found: ' + qe.getMessage());
} catch(DmlException de) {
    // DML failed - get field and message details
    for(Integer i = 0; i < de.getNumDml(); i++) {
        System.debug(de.getDmlMessage(i));
        System.debug(de.getDmlFields(i));
    }
} catch(Exception e) {
    // Catch-all for unexpected errors
    System.debug('Unexpected: ' + e.getMessage() + ' at ' + e.getStackTraceString());
}

// Custom exception class - very common in senior dev code
public class LoanProcessingException extends Exception {}

if(loan.Status__c == 'Denied') {
    throw new LoanProcessingException('Loan cannot be submitted in Denied status');
}
Never do this

Never use an empty catch block. Never catch LimitException (it cannot be caught). Never swallow exceptions silently — always log or rethrow.

What is the difference between Database.insert() and just using insert keyword? APEX

insert keyword: all-or-nothing. If even ONE record fails validation, the entire list rolls back and a DmlException is thrown.

Database.insert(list, allOrNone): when allOrNone=false, partial success is allowed. Records that pass insert succeed; failed records are captured in SaveResult. No exception is thrown for individual failures.

// Using Database.insert with partial DML - production pattern
List<Database.SaveResult> results = Database.insert(contactList, false);

for(Integer i = 0; i < results.size(); i++) {
    if(!results[i].isSuccess()) {
        for(Database.Error err : results[i].getErrors()) {
            System.debug('Failed: ' + contactList[i].LastName 
                + ' Reason: ' + err.getMessage()
                + ' Fields: ' + err.getFields());
        }
    } else {
        System.debug('Inserted: ' + results[i].getId());
    }
}
When to use which

Use allOrNone=false (Database.insert) when processing bulk records where partial failure is acceptable — like mass data loads, batch jobs. Use plain insert when all-or-nothing consistency is required, like financial transactions.

What are @AuraEnabled, @InvocableMethod, @RemoteAction, @future annotations — explain each? APEX
  • @AuraEnabled — exposes method to LWC/Aura components. Add cacheable=true only on read-only methods — enables client-side caching. Cannot be used on methods that do DML with cacheable=true.
  • @AuraEnabled(cacheable=true) — required for @wire adapter in LWC. Returns cached data without server call on subsequent requests.
  • @InvocableMethod — exposes Apex method to Flow, Process Builder. Parameters must use List<T>. Required to run in bulk for Flow interviews.
  • @future — runs method asynchronously in separate transaction. Cannot accept sObject parameters (only primitive types or collections of primitives). Good for callouts from triggers and Mixed DML.
  • @RemoteAction — legacy annotation for Visualforce JavaScript Remoting. Rarely used in new development.
  • @TestVisible — makes private method accessible to test class without changing access modifier.
  • @ReadOnly — allows querying up to 1,000,000 rows in a single transaction. Only for reporting.
@AuraEnabled(cacheable=true)
public static List<Account> getAccounts(String nameFilter) {
    return [SELECT Id, Name FROM Account WHERE Name LIKE :('%'+nameFilter+'%')];
}

@InvocableMethod(label='Send Loan Approval Email' description='Sends email to borrower')
public static void sendApprovalEmail(List<Id> loanIds) {
    // Flow calls this - must be List parameter
}

// Abstract class - use when sharing partial implementation
public abstract class LoanProcessor {
    public abstract void processLoan(Loan__c loan);  // must override
    public void logActivity(String msg) {  // shared implementation
        System.debug('[LoanProcessor] ' + msg);
    }
}
public class FHALoanProcessor extends LoanProcessor {
    public override void processLoan(Loan__c loan) { /* FHA logic */ }
}

// Interface - use when defining a contract with no shared code
public interface IDocumentSender {
    Boolean sendDocument(Id recordId, String templateName);
}
public class DocuSignSender implements IDocumentSender {
    public Boolean sendDocument(Id recordId, String templateName) {
        // DocuSign implementation
    }
}
Rule of thumb

Use abstract class when subclasses share some common behavior. Use interface when you want to define a contract that multiple unrelated classes can fulfill — like a DocuSign sender, an email sender, and an SMS sender all implementing ISender.

How do you write a test class properly — @isTest, @TestSetup, Test.startTest()? APEX

Test classes in Salesforce require 75% overall code coverage to deploy to production. But good test classes verify behavior, not just coverage.

@isTest
public class LoanTriggerTest {
    
    // @TestSetup runs ONCE before all test methods — faster, more efficient
    @TestSetup
    static void makeData() {
        Account broker = new Account(Name='Test Broker', Type='Broker');
        insert broker;
        Contact officer = new Contact(
            FirstName='Test', LastName='Officer', AccountId=broker.Id
        );
        insert officer;
    }
    
    @isTest
    static void testLoanApproval() {
        Contact officer = [SELECT Id FROM Contact LIMIT 1];
        Loan__c loan = new Loan__c(
            LoanAmount__c = 250000, Status__c = 'Application', LoanOfficer__c = officer.Id
        );
        
        // Test.startTest() resets governor limits for the test scope
        Test.startTest();
        insert loan;
        Test.stopTest(); // Forces async processes (future, batch) to complete
        
        // Re-query to check trigger effects
        Loan__c updatedLoan = [SELECT Id, Status__c FROM Loan__c WHERE Id = :loan.Id];
        System.assertEquals('Processing', updatedLoan.Status__c, 'Status should be Processing after insert');
    }
    
    @isTest
    static void testBulkLoanInsert() {
        // Always test with 200 records — simulates real bulk trigger
        List<Loan__c> loans = new List<Loan__c>();
        for(Integer i = 0; i < 200; i++) {
            loans.add(new Loan__c(LoanAmount__c = 100000 + i, Status__c = 'Application'));
        }
        Test.startTest();
        insert loans;
        Test.stopTest();
        System.assertEquals(200, [SELECT COUNT() FROM Loan__c]);
    }
}
Key rules

1. Always use Test.startTest/stopTest to get fresh limits and complete async. 2. Always test bulk (200 records). 3. Use System.assertEquals(expected, actual, 'message') — the message tells you what failed. 4. Use @TestSetup for shared data — it runs once and is rolled back between test methods. 5. SeeAllData=false by default (test data isolation). Never use SeeAllData=true unless absolutely necessary.
