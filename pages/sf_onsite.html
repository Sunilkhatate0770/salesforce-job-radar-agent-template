Explain Data Cloud basics (Ingestion, Modeling, Identity Resolution).DATA CLOUD

Data Cloud is a hyperscale data platform. Key pillars:

  • Ingestion: Bringing data in via Connectors (S3, Salesforce, Mulesoft, SDK). Data lands in a Data Lake Object (DLO).
  • Modeling: Mapping DLOs to a standardized CIM (Cloud Information Model) to create Data Model Objects (DMO).
  • Identity Resolution: Using Match Rules (fuzzy/exact) and Reconciliation Rules to unify multiple records into a single 'Unified Individual'.
  • Activation: Sending segments or insights back to Salesforce or external endpoints.
Scenario: Sharing records from Sales Rep to Senior Manager while skipping the Junior Manager.SECURITY

Standard **Role Hierarchy** won't work here because it's strictly vertical. To achieve this:

  1. OWD: Set to Private.
  2. Role Hierarchy: Keep them in the same branch, but disable 'Grant Access Using Hierarchies' on the object (if possible) or use a more refined approach.
  3. Solution: Use Criteria-Based Sharing Rules or Manual Sharing via Apex to explicitly share the Sales Rep's records with a **Public Group** that includes the Senior Manager but excludes the Junior Manager.
How do you call an integration from a Trigger (Sync vs Async)?INTEGRATION

Synchronous callouts are NOT allowed from triggers. You must use Asynchronous methods:

  • @future(callout=true): Simple, but cannot pass SObjects.
  • Queueable Apex: Better for complex logic and allows passing SObject records.
  • Platform Events: Publish an event from the trigger; have an external system or an async trigger subscriber handle the callout.

Key Constraint: Never perform DML before a callout in the same transaction.

How do you communicate between unrelated LWC components? LMS vs LMC.LWC

LMS (Lightning Message Service): The engine/service used to communicate across Visualforce, Aura, and LWC.

LMC (Lightning Message Channel): The metadata file (e.g., SampleMessageChannel.messageChannel-meta.xml) that defines the structure of the message.

You define the LMC and then use the LMS APIs (publish/subscribe) in your JS components.

Communication between a standard record page and a custom LWC.LWC

If the standard record page and custom LWC need to talk:

  • Refresh Data: Use getRecordNotifyChange(recordIds) in LWC to force the standard record page to refresh its cache after the LWC performs an update.
  • Capture standard events: Standard record pages don't usually dispatch custom events to LWCs, so we rely on Lightning Message Service (LMS) if both components are on the same page.
SOQL: Fetch Account (2 fields) and Contact (3 fields) from a Task record.CODING
// Use polymorphic fields (WhatId and WhoId)
SELECT Id, 
       What.Name, What.Type, // For Account (WhatId)
       Who.Name, Who.Email, Who.Title // For Contact (WhoId)
FROM Task 
WHERE Id = :taskId

Note: Since What and Who are polymorphic, you must use TYPEOF if you need specific fields that are not in the shared set, or just use basic relationship fields if they are common.

Scenario: Lookup vs Master-Detail on a custom 'Loan' object.DATA MODEL
  • Master-Detail: Use if the Loan MUST have an Account, access is controlled by the Account, and if the Account is deleted, the Loan should be deleted (Cascade Delete).
  • Lookup: Use if a Loan can exist without an Account, or if different security/sharing is needed for Loans compared to Accounts.

Pro-Tip: You cannot have a Master-Detail on a Standard Object if the child is also a Standard Object (except specific ones like Account-Contact which is a special lookup).
