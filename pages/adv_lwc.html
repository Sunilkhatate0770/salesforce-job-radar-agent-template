How do you optimise a slow LWC component — what are the common performance problems? ADVANCED LWC▼

Common LWC performance problems and fixes:

  • Too many wire calls on load — multiple wire adapters firing simultaneously. Fix: lazy load non-critical data using imperative calls triggered by user scroll or tab switch.
  • Re-renders on every property change — computed getters that return new object/array references each time. Fix: cache computed values, only recalculate when source data changes.
  • Large datatable without pagination — rendering 1000+ rows crashes browsers. Fix: implement server-side pagination with OFFSET or cursor-based pagination.
  • Calling refreshApex unnecessarily — triggers a server round trip. Fix: only refresh when actual DML happened. Use optimistic UI updates for better perceived performance.
  • Event listeners not cleaned up — memory leaks over time. Fix: always unsubscribe LMS, clear intervals/timeouts in disconnectedCallback.
// Problem: getter creates new array every render = unnecessary re-renders
get approvedLoans() {
    return this.loans.filter(l => l.Status__c === 'Approved'); // new array every call
}

// Fix: cache the computed value, recalculate only when loans changes
_approvedLoans = [];
set loans(value) {
    this._loans = value;
    this._approvedLoans = value ? value.filter(l => l.Status__c === 'Approved') : [];
}
get loans() { return this._loans; }
get approvedLoans() { return this._approvedLoans; } // stable reference

// Pagination pattern
currentPage = 1;
pageSize = 20;

get pagedLoans() {
    const start = (this.currentPage - 1) * this.pageSize;
    return this.allLoans.slice(start, start + this.pageSize);
}

handleNextPage() { this.currentPage++; }

What is the Shadow DOM in LWC — how does it affect CSS styling? ADVANCED LWC▼

Shadow DOM is a web standard that encapsulates a component's internal DOM and CSS. It creates a boundary — styles defined inside a component do NOT leak out to the parent, and parent styles do NOT reach inside a child component. LWC uses a synthetic Shadow DOM (and native in some contexts).

  • CSS scoping: Each component's .css file only applies to elements in THAT component's template. You cannot style a child component's internals from a parent's CSS.
  • :host selector: Styles the root element of the component itself from within the component's CSS file.
  • CSS custom properties (variables): The only way to style across shadow boundaries. Parent defines a variable, child uses it.
  • SLDS tokens: Salesforce's design system uses CSS variables for this exact reason — they cross shadow boundaries.
/* Parent component CSS - CANNOT style inside child */
/* This does NOT work: */
c-loan-card .title { color: red; } /* blocked by shadow DOM */

/* SOLUTION: CSS custom properties cross the boundary */
/* In parent CSS: */
c-loan-card {
    --loan-title-color: red;   /* define the variable on the host */
}

/* In child (loanCard.css): */
.title {
    color: var(--loan-title-color, black); /* use the variable with fallback */
}

/* :host - style the component's root element from inside */
:host {
    display: block;
    border-radius: 8px;
    overflow: hidden;
}
:host(.highlighted) { border: 2px solid blue; } /* when parent adds class to component */

How does Lightning Data Service (LDS) work — when do you use it vs Apex?ADV LWC▼

LDS is Salesforce's client-side data caching layer. It shares one cache across all LWC/Aura components on the page — if two components query the same record, only one network call is made.

// getRecord — LDS wire adapter (uses LDS cache)
import { getRecord, getFieldValue, updateRecord, deleteRecord } from 'lightning/uiRecordApi';
import { refreshApex } from '@salesforce/apex';
import LOAN_NAME from '@salesforce/schema/Loan__c.Name';
import LOAN_STATUS from '@salesforce/schema/Loan__c.Status__c';

@wire(getRecord, { recordId: '$recordId', fields: [LOAN_NAME, LOAN_STATUS] })
loan;

// Update via LDS — no Apex needed, respects FLS automatically
async handleSave() {
  const fields = { Id: this.recordId, Status__c: 'Approved' };
  try {
    await updateRecord({ fields });
    // LDS automatically invalidates cache — all other components refresh
  } catch (e) { this.showError(e); }
}

How do you use Lightning Message Service (LMS) for cross-component communication?ADV LWC▼

LMS is the modern replacement for Aura application events and PubSub patterns. Works across LWC, Aura, and Visualforce components on the same page.

// 1. Create message channel: force-app/main/default/messageChannels/LoanChannel__c.messageChannel-meta.xml
// <LightningMessageChannel xmlns="..."><masterLabel>Loan Channel</masterLabel></LightningMessageChannel>

// 2. Publisher component
import { LightningElement, wire } from 'lwc';
import { publish, MessageContext } from 'lightning/messageService';
import LOAN_CHANNEL from '@salesforce/messageChannel/LoanChannel__c';

export default class LoanList extends LightningElement {
  @wire(MessageContext) messageContext;

  handleLoanSelect(event) {
    publish(this.messageContext, LOAN_CHANNEL, {
      loanId: event.detail.loanId,
      loanName: event.detail.loanName,
      action: 'SELECTED'
    });
  }
}

// 3. Subscriber component (anywhere on same Lightning page)
import { subscribe, unsubscribe, MessageContext } from 'lightning/messageService';
import LOAN_CHANNEL from '@salesforce/messageChannel/LoanChannel__c';

export default class LoanDetail extends LightningElement {
  @wire(MessageContext) messageContext;
  subscription = null;

  connectedCallback() {
    this.subscription = subscribe(this.messageContext, LOAN_CHANNEL, (msg) => {
      if (msg.action === 'SELECTED') this.loadLoan(msg.loanId);
    });
  }

  disconnectedCallback() { unsubscribe(this.subscription); }
}

import { LightningElement } from 'lwc';
import { subscribe, unsubscribe, onError } from 'lightning/empApi';

export default class LoanStatusBoard extends LightningElement {
  subscription = {};
  loans = [];

  async connectedCallback() {
    this.subscription = await subscribe(
      '/event/Loan_Status_Change__e',
      -1, // -1 = new events only; -2 = replay all retained
      (response) => {
        const payload = response.data.payload;
        this.updateLoanStatus(payload.Loan_Id__c, payload.New_Status__c, payload.Borrower_Name__c);
      }
    );
    onError(error => { console.error('Streaming error:', error); this.reconnect(); });
  }

  updateLoanStatus(loanId, status, borrower) {
    // Update the specific loan in the list without a full refresh
    this.loans = this.loans.map(loan =>
      loan.Id === loanId ? { ...loan, Status__c: status } : loan
    );
    // Show toast for important transitions
    if (status === 'Approved') {
      this.dispatchEvent(new ShowToastEvent({ title: 'Loan Approved!',
        message: borrower + ''s loan was approved', variant: 'success' }));
    }
  }

  disconnectedCallback() { unsubscribe(this.subscription); }
}
How do you implement a custom LWC data table with sorting and filtering?ADV LWC▼
// loanTable.js
import { LightningElement, track, wire } from 'lwc';
import getLoans from '@salesforce/apex/LoanController.getLoans';

export default class LoanTable extends LightningElement {
  @track loans = [];
  @track filteredLoans = [];
  sortField = 'Name'; sortDirection = 'asc'; filterText = '';

  columns = [
    { label:'Loan', fieldName:'Name', type:'text', sortable:true },
    { label:'Status', fieldName:'Status__c', type:'text', sortable:true },
    { label:'Amount', fieldName:'Amount__c', type:'currency', sortable:true },
    { label:'Date', fieldName:'CloseDate__c', type:'date', sortable:true },
  ];

  @wire(getLoans)
  wiredLoans({ data }) { if (data) { this.loans = data; this.applyFilter(); } }

  handleSort(event) {
    this.sortField = event.detail.fieldName;
    this.sortDirection = event.detail.sortDirection;
    this.applyFilter();
  }

  handleFilter(event) { this.filterText = event.target.value; this.applyFilter(); }

  applyFilter() {
    let result = [...this.loans];
    // Filter
    if (this.filterText) {
      const term = this.filterText.toLowerCase();
      result = result.filter(l => l.Name?.toLowerCase().includes(term) || l.Status__c?.toLowerCase().includes(term));
    }
    // Sort
    result.sort((a, b) => {
      const valA = a[this.sortField] ?? ''; const valB = b[this.sortField] ?? '';
      const cmp = valA > valB ? 1 : valA < valB ? -1 : 0;
      return this.sortDirection === 'asc' ? cmp : -cmp;
    });
    this.filteredLoans = result;
  }

  get totalCount() { return this.filteredLoans.length; }
}
How do you handle file uploads in LWC — upload to Salesforce Files and to external storage?ADV LWC▼
// Upload to Salesforce Files
<!-- Template -->
<lightning-file-upload label="Upload Documents" accept=".pdf,.docx"
  record-id={recordId} onuploadfinished={handleUpload}></lightning-file-upload>

// JS handler
handleUpload(event) {
  const files = event.detail.files; // [{ documentId, name, size }]
  const fileIds = files.map(f => f.documentId);
  this.dispatchEvent(new CustomEvent('filesadded', { detail: { fileIds } }));
  this.dispatchEvent(new ShowToastEvent({ title:'Uploaded', message:files.length+' file(s)', variant:'success'}));
}

// Custom file upload with progress (for large files or external storage)
async handleCustomUpload(event) {
  const file = event.target.files[0];
  const reader = new FileReader();
  reader.onload = async () => {
    const base64 = reader.result.split(',')[1];
    try {
      this.uploadProgress = 0;
      // Call Apex with base64 — max 4.5MB before base64 encoding
      const result = await uploadDocument({ fileName: file.name, base64Content: base64, recordId: this.recordId });
      this.uploadProgress = 100;
    } catch(e) { this.showError(e.body.message); }
  };
  reader.readAsDataURL(file);
}

When to use slots

Slots are for layout/container components where the parent provides content. Examples: modal dialogs, card containers, tab panels, wizard steps. The container handles layout; the parent provides the specific content.

How do you implement a wizard / multi-step form in LWC?ADV LWC▼
// loanApplicationWizard.js
import { LightningElement, track } from 'lwc';

export default class LoanApplicationWizard extends LightningElement {
  @track currentStep = 1;
  @track loanData = { borrowerInfo: {}, loanDetails: {}, documents: [] };
  totalSteps = 4;
  steps = ['Borrower Info','Loan Details','Documents','Review & Submit'];

  get progressPercent() { return Math.round((this.currentStep / this.totalSteps) * 100); }
  get isFirstStep() { return this.currentStep === 1; }
  get isLastStep() { return this.currentStep === this.totalSteps; }
  get currentStepLabel() { return this.steps[this.currentStep - 1]; }

  // Each step fires an event with its data when valid
  handleStepComplete(event) {
    const stepData = event.detail;
    if (this.currentStep === 1) this.loanData.borrowerInfo = stepData;
    if (this.currentStep === 2) this.loanData.loanDetails = stepData;
    if (this.currentStep === 3) this.loanData.documents = stepData.fileIds;
    if (this.currentStep < this.totalSteps) this.currentStep++;
  }

  handleBack() { if (this.currentStep > 1) this.currentStep--; }

  async handleSubmit() {
    try {
      await submitLoanApplication({ loanData: JSON.stringify(this.loanData) });
      this.dispatchEvent(new ShowToastEvent({ title:'Application Submitted', variant:'success' }));
    } catch(e) { this.showError(e); }
  }
}
How do you use dynamic components in LWC — lwc:component and lazy loading?ADV LWC▼
// Dynamic component — component type determined at runtime
// loanActionPanel.html
<template>
  <lwc:component lwc:is={dynamicComponent} loan-id={recordId} onactioncomplete={handleComplete}></lwc:component>
</template>

// loanActionPanel.js
import { LightningElement, api } from 'lwc';

export default class LoanActionPanel extends LightningElement {
  @api recordId;
  @api actionType; // 'approve', 'decline', 'request-docs'

  // Lazy load the component based on actionType
  get dynamicComponent() {
    switch(this.actionType) {
      case 'approve':       return import('c/loanApprovalForm');
      case 'decline':       return import('c/loanDeclineForm');
      case 'request-docs':  return import('c/documentRequestForm');
      default: return null;
    }
  }
  handleComplete(event) {
    this.dispatchEvent(new CustomEvent('actioncomplete', { detail: event.detail }));
  }
}
Dynamic import benefit

The component bundle is only downloaded from Salesforce servers when needed. This reduces initial page load time — a loan officer who never declines loans never downloads the decline form code.
