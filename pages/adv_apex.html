What is the difference between Database.Stateful and without Database.Stateful in Batch Apex? When does it matter? ADVANCED APEXâ–¼

By default, each execute() chunk of a batch job runs in a fresh, isolated transaction. All instance variables are reset between chunks. Database.Stateful changes this â€” instance variables are preserved across all execute() calls for the lifetime of the batch job.

FeatureWithout Stateful (default)With Database.Stateful
Instance variables between chunksReset to null/defaultPreserved
Memory usageLowerHigher (state held in memory)
Use forIndependent chunk processingCounters, error lists, accumulators
Heap limit12 MB per chunk12 MB but state counts toward it
public class LoanReportBatch implements Database.Batchable<SObject>, Database.Stateful {

    // These variables survive across ALL execute() calls
    public Integer totalProcessed = 0;
    public Integer totalApproved = 0;
    public List<String> errorMessages = new List<String>();

    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator('SELECT Id, Status__c, LoanAmount__c FROM Loan__c');
    }

    public void execute(Database.BatchableContext bc, List<SObject> scope) {
        for(Loan__c loan : (List<Loan__c>) scope) {
            totalProcessed++; // persists across chunks - won't reset!
            if(loan.Status__c == 'Approved') totalApproved++;
        }
    }

    public void finish(Database.BatchableContext bc) {
        // totalProcessed now has the REAL total across ALL chunks
        System.debug('Processed: ' + totalProcessed + ', Approved: ' + totalApproved);
        EmailService.sendReport(totalProcessed, totalApproved, errorMessages);
    }
}
Without Stateful â€” the bug trap

Without Stateful, if you have a counter variable and increment it in execute(), it resets to 0 for every chunk. Your finish() method sees 0 or only the count from the last chunk â€” not the true total. This is a very common production bug for developers who don't understand batch state.

Explain Apex CPU time optimisation â€” what consumes CPU and how do you reduce it? ADVANCED APEXâ–¼

CPU time limit is 10,000ms synchronous / 60,000ms async. It measures actual Apex execution time â€” not waiting for SOQL or callouts (those don't count toward CPU). The most common causes of CPU limit errors:

  • String concatenation in loops â€” String is immutable in Apex. Concatenating in a loop creates a new String object every iteration. Use List and String.join() instead.
  • Nested loops over large collections â€” O(nÂ²) complexity. Replace inner loop lookups with a Map for O(1).
  • JSON serialization of large objects â€” JSON.serialize() on deep object graphs is expensive. Serialize only what you need.
  • Regex operations in loops â€” Pattern.compile() is expensive. Compile once, reuse the Matcher.
  • Complex formula fields in SOQL result loops â€” Avoid querying formula fields when you don't need them.
// BAD - String concatenation in loop = new object every iteration
String result = '';
for(Loan__c l : loans) { result += l.LoanAmount__c + ','; } // CPU killer

// GOOD - List + join = one String operation at end
List<String> parts = new List<String>();
for(Loan__c l : loans) { parts.add(String.valueOf(l.LoanAmount__c)); }
String result = String.join(parts, ','); // single operation

// BAD - nested loop = O(nÂ²)
for(Loan__c l : loans) {
    for(Account a : accounts) { // checks every account for every loan
        if(a.Id == l.AccountId__c) { l.BrokerName__c = a.Name; }
    }
}

// GOOD - Map lookup = O(1) per loan
Map<Id, Account> accMap = new Map<Id, Account>(accounts);
for(Loan__c l : loans) {
    if(accMap.containsKey(l.AccountId__c)) l.BrokerName__c = accMap.get(l.AccountId__c).Name;
}

// BAD - compile regex every iteration
for(String s : inputs) { if(Pattern.compile('\\d{3}-\\d{2}-\\d{4}').matcher(s).matches()) {...} }

// GOOD - compile once, reuse
Pattern ssnPattern = Pattern.compile('\\d{3}-\\d{2}-\\d{4}');
for(String s : inputs) { if(ssnPattern.matcher(s).matches()) {...} }
What is Apex Managed Sharing â€” when and how do you implement it? ADVANCED APEXâ–¼

Apex Managed Sharing allows you to programmatically grant record access beyond what OWD and sharing rules provide, using a custom RowCause. This is more powerful than manual sharing because it persists across record ownership changes and can be fully automated.

// Every custom object gets a __Share object automatically
// Loan__c â†’ Loan__Share
// Fields: ParentId (record), UserOrGroupId (who gets access), AccessLevel, RowCause

public class LoanSharingService {

    // RowCause must be defined as a Sharing Reason on the object in setup
    private static final String ROW_CAUSE = Schema.Loan__Share.RowCause.BrokerAccess__c;

    public static void shareLoanWithBroker(Id loanId, Id brokerUserId) {
        Loan__Share share = new Loan__Share();
        share.ParentId = loanId;               // the loan record
        share.UserOrGroupId = brokerUserId;    // the broker user
        share.AccessLevel = 'Read';           // Read, Edit (not Full on custom)
        share.RowCause = ROW_CAUSE;            // custom reason - required for apex sharing

        Database.SaveResult sr = Database.insert(share, false);
        if(!sr.isSuccess()) {
            System.debug('Share failed: ' + sr.getErrors()[0].getMessage());
        }
    }

    public static void removeShareForBroker(Id loanId, Id brokerUserId) {
        List<Loan__Share> toDelete = [
            SELECT Id FROM Loan__Share
            WHERE ParentId = :loanId
            AND UserOrGroupId = :brokerUserId
            AND RowCause = :ROW_CAUSE
        ];
        if(!toDelete.isEmpty()) delete toDelete;
    }
}
Real project use case

In the Mortgage Origination Platform, when a loan was assigned to a broker partner, we used Apex Managed Sharing to give that broker's user Read access to the loan. When the broker was replaced, the old share was deleted and a new one created. Standard sharing rules could not handle this dynamic, per-record assignment logic.

What is Queueable chaining â€” how do you chain 3 jobs and handle failures in the chain? ADVANCED APEXâ–¼ 
// Chain: CreditCheck â†’ OFACScreening â†’ DocuSignTrigger
// Each job chains the next only if it succeeds

public class CreditCheckJob implements Queueable, Database.AllowsCallouts {
    private Id loanId;

    public CreditCheckJob(Id loanId) { this.loanId = loanId; }

    public void execute(QueueableContext ctx) {
        try {
            Loan__c loan = [SELECT Id, SSN__c, LastName__c FROM Loan__c WHERE Id = :loanId];
            CreditResult result = ExperianService.pullCredit(loan.SSN__c, loan.LastName__c);

            loan.FicoScore__c = result.ficoScore;
            loan.CreditStatus__c = result.ficoScore >= 620 ? 'Passed' : 'Failed';
            update loan;

            if(loan.CreditStatus__c == 'Passed') {
                // Chain next job only on success
                System.enqueueJob(new OFACScreeningJob(loanId));
            } else {
                // Mark loan as needing manual review - don't chain
                LoanStatusService.flagForReview(loanId, 'Credit score below minimum');
            }
        } catch(Exception e) {
            // Log error, set error status on loan - don't let exception kill silently
            ErrorLogger.log('CreditCheckJob', loanId, e.getMessage(), e.getStackTraceString());
            LoanStatusService.setError(loanId, 'Credit check failed: ' + e.getMessage());
        }
    }
}

public class OFACScreeningJob implements Queueable, Database.AllowsCallouts {
    private Id loanId;
    public OFACScreeningJob(Id loanId) { this.loanId = loanId; }
    public void execute(QueueableContext ctx) {
        // OFAC check logic here, then chain DocuSign if passes
        System.enqueueJob(new DocuSignTriggerJob(loanId));
    }
}

// To start the chain:
System.enqueueJob(new CreditCheckJob(loan.Id));
