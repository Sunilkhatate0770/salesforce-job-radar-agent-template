How do you control trigger execution order if multiple triggers exist on the same object?ARAGO▼

Salesforce does not guarantee execution order when multiple triggers exist on the same object. The only reliable solution is the one trigger per object rule.

// One trigger — acts as dispatcher only
trigger LoanTrigger on Loan__c (before insert, before update, after insert, after update, after delete) {
    new LoanTriggerHandler().run();
}

// Handler controls execution order explicitly via ordered method calls
public class LoanTriggerHandler extends TriggerHandler {
    public override void onBeforeInsert() {
        LoanValidationService.validate(Trigger.new);   // 1st
        LoanDefaultsService.setDefaults(Trigger.new);  // 2nd — order guaranteed
    }
    public override void onAfterInsert() {
        LoanNotificationService.sendWelcome(Trigger.new); // 1st
        LoanLOSService.syncToLOS(Trigger.new);            // 2nd
    }
}
Interview answer

Say: 'I enforce one trigger per object in every project. Execution order is controlled within the handler class — the order method calls are made is explicit and fully predictable.'

How do you handle bulk insert/update operations efficiently in triggers?ARAGO▼

Core rule: never put SOQL or DML inside a for loop. Always process the full Trigger.new batch collectively.

// BAD — hits 101-query limit with 101 records
for (Loan__c l : Trigger.new) {
    Account a = [SELECT Id FROM Account WHERE Id = :l.Account__c]; // SOQL IN LOOP
    a.Total_Loans__c++; update a; // DML IN LOOP
}

// GOOD — ONE SOQL + ONE DML for entire batch
Set<Id> accIds = new Set<Id>();
for (Loan__c l : Trigger.new) if (l.Account__c != null) accIds.add(l.Account__c);

Map<Id,Account> accMap = new Map<Id,Account>(
    [SELECT Id, Total_Loans__c FROM Account WHERE Id IN :accIds]);

for (Loan__c l : Trigger.new)
    if (accMap.containsKey(l.Account__c)) accMap.get(l.Account__c).Total_Loans__c++;

update accMap.values(); // one DML for all accounts
  • Use Set to collect unique IDs
  • Use Map<Id,SObject> for O(1) lookup
  • Single SOQL outside loop
  • Single DML outside loop on full collection

How can you prevent recursion in triggers?ARAGO▼

Pattern 1 — Static boolean (simplest):

public class TriggerHelper { public static Boolean loanTriggerRan = false; }
trigger LoanTrigger on Loan__c (after update) {
    if (TriggerHelper.loanTriggerRan) return;
    TriggerHelper.loanTriggerRan = true;
    LoanTriggerHandler.onAfterUpdate(Trigger.new, Trigger.oldMap);
}

Pattern 2 — Set of processed IDs (more precise):

public class TriggerHelper { public static Set<Id> processedIds = new Set<Id>(); }
trigger LoanTrigger on Loan__c (after update) {
    List<Loan__c> unprocessed = new List<Loan__c>();
    for (Loan__c l : Trigger.new)
        if (!TriggerHelper.processedIds.contains(l.Id)) { unprocessed.add(l); TriggerHelper.processedIds.add(l.Id); }
    if (!unprocessed.isEmpty()) LoanTriggerHandler.onAfterUpdate(unprocessed, Trigger.oldMap);
}
Why it works

Static variables persist for the entire transaction but reset between transactions. A workflow re-entry sees the same static variable — the guard stops the loop.

What is a Savepoint and how does rollback work?ARAGO▼
Savepoint sp = Database.setSavepoint();
try {
    Loan__c l = new Loan__c(Status__c='Processing'); insert l;
    LoanDocument__c d = new LoanDocument__c(Loan__c=l.Id); insert d;
    if (failureCondition) throw new LoanException('Failed');
} catch (LoanException e) {
    Database.rollback(sp); // undoes BOTH inserts above
    insert new Error_Log__c(Message__c=e.getMessage()); // still works after rollback
}
Callouts + Savepoints

Cannot callout after DML in same tx. Cannot use savepoint after callout. Order must be: callout → DML, or use async for callout.

How do you design large-scale async processing?ARAGO▼
// Scheduled → Batch → Queueable chain
public class NightlySync implements Schedulable {
    public void execute(SchedulableContext ctx) { Database.executeBatch(new LoanSyncBatch(), 200); }
}
System.schedule('Nightly Sync', '0 0 2 * * ?', new NightlySync());

public class LoanSyncBatch implements Database.Batchable<SObject>, Database.Stateful, Database.AllowsCallouts {
    public Integer processed=0, failed=0;
    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator([SELECT Id FROM Loan__c WHERE Status__c='Active']);
    }
    public void execute(Database.BatchableContext bc, List<SObject> scope) {
        for (Loan__c l : (List<Loan__c>)scope) {
            try { LOSService.sync(l); processed++; }
            catch (Exception e) { failed++; logError(l.Id,e); }
        }
    }
    public void finish(Database.BatchableContext bc) {
        System.enqueueJob(new PostSyncQueueable(processed, failed)); // chain notification
    }
}
  • Database.Stateful — persist counts across chunks
  • AllowsCallouts — enable HTTP in execute()
  • Lower scope for callout-heavy batches (10-50)
  • finish() — post-processing, notifications, chaining

What happens if a child dispatches an event before the parent is rendered?MS ROUND 1▼

The event will not be captured by the parent. Events propagate through an existing, connected DOM tree — if the parent isn't in the DOM yet when the event fires, there is nothing to receive it.

// PROBLEM — event may fire before parent listener is attached
connectedCallback() {
    this.dispatchEvent(new CustomEvent('dataready', { detail: { id: this.recordId } }));
}

// FIX 1 — defer to next microtask (parent guaranteed connected by then)
connectedCallback() {
    Promise.resolve().then(() => {
        this.dispatchEvent(new CustomEvent('dataready', { detail: { id: this.recordId } }));
    });
}

// FIX 2 — dispatch from renderedCallback() — parent always rendered before child
renderedCallback() {
    if (!this.initialized) {
        this.initialized = true;
        this.dispatchEvent(new CustomEvent('dataready', { detail: { id: this.recordId } }));
    }
}
Interview insight

This tests your understanding of LWC's async rendering model — the DOM tree must exist for event propagation to work. Timing matters in component trees.

How does Salesforce handle callout responses that arrive after a timeout?MS ROUND 1▼

Once the timeout is reached Salesforce marks the request failed and throws a CalloutException. Any late response from the server is completely ignored.

  • Max timeout — 120,000ms. Default is 10,000ms if not set.
  • Late response — discarded. No mechanism to receive it.
  • Side effect risk — external system may have already processed the request. Always use idempotency keys — retrying a timed-out request must not create duplicates.
try {
    req.setTimeout(15000);
    HttpResponse res = new Http().send(req);
} catch (CalloutException e) {
    if (e.getMessage().contains('Read timeout')) {
        loan.Sync_Status__c = 'Timeout_Pending';
        loan.Correlation_Id__c = correlationId;
        update loan;
    }
}
Why can the same SOQL query return different results within one transaction?MS ROUND 1▼

Within a transaction, SOQL reflects uncommitted in-memory changes. Insert or update records in Apex, and the next SOQL query sees those changes — even before commit.

List<Loan__c> before = [SELECT Id FROM Loan__c WHERE Status__c='Applied'];
Loan__c newLoan = new Loan__c(Status__c='Applied', Amount__c=100000); insert newLoan;
List<Loan__c> after = [SELECT Id FROM Loan__c WHERE Status__c='Applied']; // sees uncommitted insert

Why are callouts not allowed after DML in the same transaction?MS ROUND 1▼
  • Record locking — DML may lock records. A callout can take up to 120 seconds, holding those locks and blocking other users.
  • Inconsistent state risk — if callout never returns, DML already ran — partial committed state.
// CORRECT — callout first, then DML
HttpResponse res = new Http().send(req);

▼ Round 2 — Technical (Advanced Platform Behavior)
What happens if a Batch Apex job is aborted while an execute() is in progress?MS ROUND 2▼
  • Current execute() chunk completes — the in-progress chunk finishes its transaction.
  • No further batches run — engine stops after current chunk.
  • finish() is NOT called.
Why does a before insert trigger not have record IDs but after insert does?MS ROUND 2▼

IDs are generated by the database engine when a record is saved. Before insert runs before the save — no ID exists yet.

An LWC has a @track object. If you reassign the same reference, will the UI re-render?MS ROUND 2▼

No. LWC tracks changes by object reference. Same reference = no change detected = no re-render.
