Explain the basic setup steps for Agentforce.AGENTFORCE

Setting up Agentforce involves 5 primary steps:

  1. Enable Einstein: Navigate to Setup — Einstein Setup and turn on Einstein.
  2. Create Agent: Go to Agent Builder and create a new Agent (e.g., Service Agent or custom).
  3. Define Topics: Create 'Topics' which represent the areas of expertise for your agent.
  4. Add Actions: Assign standard or custom actions (Flows, Apex @InvocableMethod, Prompt Templates) to specific topics.
  5. Activate & Deploy: Activate the agent and deploy it to a channel like Experience Cloud, Messaging, or Slack.
Interview Pro-Tip

If asked about 'Atlas', mention that it's the reasoning engine that uses the ReAct (Reason + Act) loop to evaluate user intent against available topics and actions. This shows you understand the 'brain' of the agent, not just the setup.

What are the different types of agents in Agentforce?AGENTFORCE

Agents are generally categorized by their primary role and interaction model:

  • Service Agent: Focused on customer support, resolving cases, and answering FAQ.
  • Sales Agent: Handles lead qualification, SDR outreach, and appointment scheduling.
  • Information Provider: Acts as a knowledge base assistant, summarizing articles and data.
  • Personal Assistant (Copilot): Employee-facing agent to help with internal tasks like record creation, summarization, and email drafting.
Explain 'Actions' in Agentforce and give examples.AGENTFORCE

Actions are the 'tools' the agent uses to perform tasks. They can be:

  • Flow Actions: Calling a Headless Flow to perform logic or updates.
  • Apex Actions: Using @InvocableMethod to call custom Apex (e.g., external API calls).
  • Prompt Template Actions: Using Prompt Builder to generate specific text responses.
  • Standard Actions: Pre-built actions like 'Summarize Record' or 'Find Records'.
How do you set up 'Topics' in Agentforce?AGENTFORCE

Topics provide the reasoning engine with context. You set them up by:

  1. Defining a Name and Description (Crucial: the LLM uses the description to decide if this topic matches user intent).
  2. Providing Instructions: Guidelines on how the agent should behave within this topic.
  3. Assigning Actions: Mapping which specific tools are relevant to this topic.
Explain RAG and Hallucination in the context of AI.AI CONCEPTS

RAG (Retrieval-Augmented Generation): The process of retrieving relevant data from your Salesforce org (like record data or Knowledge articles) and injecting it into the prompt. This 'grounds' the AI in your specific data.

Hallucination: When an AI model generates confident but false or fabricated information because it lacks specific context. RAG is the primary solution to prevent hallucinations in Salesforce.

Explain LWC Variables (@api, @track, non-decorated).LWC

LWC uses three main types of property reactivity:

  • @api: Public properties. Used to pass data from parent to child. Making a property public makes it reactive to parent changes.
  • @track: Private reactive properties for deep tracking. Necessary for observing changes inside an object or array (e.g., this.obj.name = 'X').
  • Non-decorated: Private reactive properties. Since Spring '20, all private properties are reactive if you re-assign the whole value (e.g., this.count = 5).
Coding: Show industry data using imperative apex in a data table with an industry filter.CODING
// JS
import { LightningElement, track } from 'lwc';
import getAccountsByIndustry from '@salesforce/apex/AccountController.getAccountsByIndustry';

export default class IndustryTable extends LightningElement {
    @track accounts = [];
    selectedIndustry = 'Technology';

    handleIndustryChange(event) {
        this.selectedIndustry = event.target.value;
        this.loadAccounts();
    }

    loadAccounts() {
        getAccountsByIndustry({ industry: this.selectedIndustry })
            .then(result => { this.accounts = result; })
            .catch(error => { console.error(error); });
    }
}

// APEX
@AuraEnabled
public static List getAccountsByIndustry(String industry) {
    return [SELECT Id, Name, Industry, AnnualRevenue FROM Account WHERE Industry = :industry LIMIT 50];
}
How do you communicate between non-connected (sibling) components?LWC

Use Lightning Message Service (LMS). Steps:

  1. Create a Lightning Message Channel (XML file).
  2. In Publisher: Use publish(this.messageContext, CHANNEL, payload).
  3. In Subscriber: Use subscribe(this.messageContext, CHANNEL, (msg) => { ... }).

Previously, pubsub was used, but LMS is now the standard as it works across LWC, Aura, and Visualforce.

Explain LDS (Lightning Data Service) in LWC.LWC

LDS is a client-side cache and data management layer. It allows you to interact with Salesforce data without writing Apex. Benefits include:

  • Automatic Caching: Data is cached across components.
  • Shared Cache: If one component updates a record, all other LDS-linked components update automatically.
  • Wire Adapters: getRecord, createRecord, updateRecord.
  • Base Components: lightning-record-form, lightning-record-view-form.
Coding: Write a trigger to prevent duplicate contacts using Email.CODING
trigger ContactDuplicatePreventer on Contact (before insert, before update) {
    Set emailSet = new Set();
    for (Contact c : Trigger.new) {
        if (c.Email != null) emailSet.add(c.Email);
    }
    
    if (!emailSet.isEmpty()) {
        List existing = [SELECT Email FROM Contact WHERE Email IN :emailSet];
        Set existingEmails = new Set();
        for (Contact ec : existing) existingEmails.add(ec.Email);
        
        for (Contact c : Trigger.new) {
            if (existingEmails.contains(c.Email)) {
                c.Email.addError('A contact with this email already exists.');
            }
        }
    }
}

Deep Dive: Batch Apex, Database.Stateful, and Database.AllowsCallouts.APEX
  • Batch Apex: Implements Database.Batchable with 3 methods: start(), execute(), finish().
  • Database.Stateful: By default, batch chunks are stateless. Implement this to maintain variable values (like counters) across different execute() chunks.
  • Database.AllowsCallouts: Mandatory if you need to perform HTTP requests within the execute() method.
Scenario: Google Drive integration using Queueable Apex.INTEGRATION

To sync Salesforce files to Google Drive:

  1. Trigger on ContentVersion (After Insert) captures the file.
  2. Enqueue a Queueable class implementing Database.AllowsCallouts.
  3. In execute(), fetch the file Blob and perform a POST request to Google Drive API using a Named Credential for OAuth.
  4. Update the original record in Salesforce with the Google Drive File ID returned.
Security: How to share records if OWD is Private?SECURITY

If OWD is Private, only the owner and those above in the Role Hierarchy can see records. To expand access:

  • Sharing Rules: Share to Roles, Public Groups based on criteria or owner.
  • Manual Sharing: User manually grants access to another user (Classic/via Apex).
  • Apex Sharing: Programmatically create __Share records for complex logic.
  • Team Sharing: Use Account/Case/Opportunity Teams to grant access.
Deep Knowledge

Remember that 'View All' and 'Modify All' permissions on an object (via Profile or Permission Set) override all sharing rules and OWD. Also, Apex triggers and most classes run in System Mode (ignoring sharing) unless you explicitly use with sharing or inherited sharing.
